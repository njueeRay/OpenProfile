## 二、PM 版本哲学提案

**理念 A（传统）：** Major.Minor.Patch，Major 代表重大重构

**理念 B（推荐）：** 增量发布，拒绝大版本跃进。每个 Minor 版本聚焦一个可交付主题，发布频率 > 变更量

PM 建议采用理念 B，并提出三级粒度规则：

| 级别 | 触发条件 | 示例 |
|------|---------|------|
| Patch | Bug 修复、内容更新、小改动 | `v4.0.1` |
| Minor | 一个有主题的功能 Sprint（1-3 天工作量） | `v4.2.0` |
| Major | 架构级变更（框架迁移、全站重构） | `v5.0.0` |

## 三、三层版本体系

| 层级 | 内容 | 版本号位置 |
|------|------|-----------|
| L1 项目版本 | `vX.Y.Z`，对应 CHANGELOG + GitHub Tag | `CHANGELOG.md` |
| L2 Playbook 版本 | `Playbook vX.Y`，团队规范手册 | `team-playbook.md` 头部 |
| L3 Agent 版本 | 各 Agent `vX.Y`，能力和权限更新 | 各 Agent 文件 `agentVersion` 字段 |
