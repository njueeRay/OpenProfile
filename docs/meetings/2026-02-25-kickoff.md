## 各成员发言摘要

**profile-designer：** 三仓库职责划分清晰，OpenProfile 必须是「双重定位」——既是用户实际在用的配置仓库，又是教学模板，"我用的就是这套流程"才有说服力，纯模板无灵魂。

**researcher：** 简历站技术选型 Astro + TypeScript。理由：TS 原生友好；静态导出直推 GitHub Pages 零运维成本（不依赖 Vercel）；有现成暗色终端风主题；相比 Hugo/Jekyll 对 TS 开发者更自然。

**content-writer：** OpenProfile 的 README 叙事主角是"这套 AI 工作流"，首屏应为「这是 njueeRay 实际在用的主页配置，用 AI Agent 构建」，badge 链接直通 njueeray.github.io。

**qa-reviewer：** 单一信息源原则——个人信息只在 OpenProfile 的配置文件中维护，其他仓库通过动态引用获取，避免三处维护分裂。

## 团队决议

### 仓库架构

```
访客
 ├─ github.com/njueeRay                    [Profile README 入口]
 │   njueeRay/njueeRay — 精简门脸·导流
 │
 ├─ github.com/njueeRay/OpenProfile        [开源工程，本项目]
 │   教学模板 + Agent 配置原点
 │   README → njueeray.github.io
 │
 └─ njueeray.github.io                    [个人简历主站]
     Astro 静态站 · 暗色终端风 · 完整简历
```

| 仓库 | 定位 | 技术栈 | 更新频率 |
|------|------|--------|---------|
| `njueeRay/njueeRay` | GitHub Profile 入口 | Markdown + GitHub Actions | 低频（自动化更新） |
| `njueeRay/OpenProfile` | 教学模板+Agent配置 | Markdown + YAML | 中频 |
| `njueeRay/njueeray.github.io` | 个人简历主站 | Astro 4.x + TypeScript | 中频 |

- **简历站**：Astro 4.16.18 + TypeScript → 静态导出 → GitHub Pages（5.x 有 ohash 依赖问题，锁定 4.x）
