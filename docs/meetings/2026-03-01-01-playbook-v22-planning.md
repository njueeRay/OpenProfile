## PM：文档治理 + 命名序号

我对「文档治理」最关心的不是规范的数量，而是规范的**可执行性**。

Ray 提到了两件具体的事：

**第一件：会议文件需要序号。**

现在的命名格式是 `YYYY-MM-DD-<类型>.md`。问题是：同日两次会议怎么区分？现在用后缀来区分（比如 `-planning` vs `-retrospective`），但如果类型相同或不确定，就容易混。

我的提案：**`YYYY-MM-DD-NN-<类型>.md`**，NN 是两位数的当日序号（01、02、03...）。

- `2026-03-01-01-playbook-v22-planning.md` ← 正在写的这篇
- `2026-03-01-02-free-brainstorm.md` ← 下午的自由讨论

这样按文件名排序就是按时间顺序，不需要人工解读。**文件名的冗余度要低，排序逻辑要显而易见。**

**第二件：文档内容里要标记序号。**

这个更简单——在每篇纪要的头部加一行 `序号：当日第 NN 次会议`。本篇已经这样做了。Playbook 里需要规范化这个格式。

**另一个文档治理建议：**  
Ray 说「维护一个稳定且可增长迭代的文档体系」——我认为这指的是文档必须有**明确的所有权**和**更新触发条件**。我在 `docs/team-playbook.md §2` 里看到了 SSOT 表格，但它只覆盖了核心文档。**把这张表扩展为「文档治理表」，加入维护责任人和过期判断标准，会让整个体系更健壮。**

## Dev：Sprint 和 Priority 的工程化

我从实现角度来说 Sprint 和 Priority。

**Sprint：** Playbook 里已经有「Sprint 规划会」和「Sprint 目标」，但没有 Sprint 的实质定义。如果一个新成员来了，他知道「要开 Sprint 规划会」，但不知道「一个 Sprint 是多长」「Sprint 结束的标志是什么」「Sprint 内发现优先级冲突怎么办」。

我建议加一个轻量的 Sprint 定义：
- **Sprint 的默认周期**：1-5 天（根据任务复杂度由 PM 决定，不强制固定）
- **Sprint 的硬约束**：进 Sprint 的任务必须有明确的 DoD；进 Sprint 的任务数必须 ≤ PM 评估的 2 天工作量以内
- **Sprint 中途不接受新任务**：除非是 P0 阻断问题；否则进入下一个 Sprint 的 Backlog

**Priority：** 现在 P0/P1/P2 在 `§3.1` 里出现，但没有独立的定义。我需要有清晰的定义，不然「P0 阻断」和「P1 高优先」之间的边界是模糊的。

**我建议的四级优先级体系（参考 MoSCoW + 工程实践）：**

| 级别 | 定义 | Sprint 处理规则 |
|------|------|----------------|
| P0 | 阻断：当前 Sprint 无法继续，或上线后系统不可用 | 立即停下所有 P1/P2，先处理 P0 |
| P1 | 高优先：当前 Sprint 必须完成 | 正常执行，不得延期 |
| P2 | 中优先：本 Sprint 尽力完成，可延期至下一 Sprint | 有余力就做 |
| P3 | 低优先 / Backlog：想做但不紧急 | 进 Backlog，不排 Sprint |

这个体系足够简单、足够明确，能解决实际冲突。

## Code Reviewer：DoD 的实质意义

DoD 在 `§10` 里已经有了，我不需要重写它。但我想说一个核心观点：

**DoD 不应该只是一个检查清单，它应该是 Sprint 开始前就写好的「验收合同」。**

现在的流程是：做完之后 PM 来核查 DoD。我建议把它提前到 Sprint 规划时：**每个任务在进入 Sprint 之前，就必须有明确的 Done = 什么**。

这一改变的价值是：
1. 发现「做不完的」任务（如果说不清楚 Done 是什么，说明任务本身没想清楚）
2. 减少 Sprint 内的方向漂移
3. 让 Code Reviewer 的工作有了标准，而不是每次都要重新讨论「这算完成了吗」

**具体操作：** Sprint 规划时，每个任务条编写格式改为：

```
- [ ] 任务描述 [P0/P1/P2] → Done = <具体的验收标准>
```

这一格式变更不增加复杂度，但能显著提升执行质量。

## Researcher：Git Worktree 是否进 Playbook

这是今天最有趣的一个问题——**Git Worktree 是工具还是方法论？**

Git Worktree 本身是 Git 的一个功能，不是方法论。但是，围绕 Worktree 的一套规范——何时开新 Worktree、如何命名分支、如何汇报给主窗口、如何合并和清理——**这是方法论**。

我们在 OpenProfile 项目里已经把这套流程跑通了，并且在 copilot-instructions 里有记录。问题是：这套记录是「项目特有的」还是「可迁移的通用规范」？

我的判断：**大部分是可迁移的**。分支命名约定、worktree 生命周期管理、汇报模板——这些跟项目内容无关，只跟 Git 操作习惯有关。

但有一个部分是项目相关的：「哪些任务适合开 Worktree 并行」——这取决于项目的任务颗粒度，不能写进通用 Playbook。

**结论建议：写入 Playbook，但只写规范，不写"在什么情况下开 Worktree"（这个由 Brain 在每个项目中根据具体情况判断）。**

## Profile Designer：copilot-instructions 的迁移边界

我最关心的是 copilot-instructions 在跨项目迁移时的体验。

Ray 的洞察非常准确：copilot-instructions 里有两类内容——

**固定结构（带走）：**
- Agent 团队分工表（含能力快照）
- 协作期望（哪些能自主、哪些需确认）
- 迭代完成检查项（DoD Checklist）
- 版本哲学

**可替换内容（到新项目时修改）：**
- 个人/项目身份信息（name, repo, stack 等）
- 技术选型决策（项目特有的版本锁定）
- 已决定的设计选择（项目特有的 UI/架构决策）
- 当前迭代状态（新项目从零开始）
- 文件结构说明（项目特有）

**问题是：现在的 copilot-instructions.md 里，这两类内容是混在一起的。** 新接手者不知道哪些可以直接用，哪些需要清空重写。

我的建议：**在 copilot-instructions.md 里用注释标记出「固定区块」和「按项目替换的区块」。** 或者在 Playbook 里列出这张表，让 Brain 在迁移时知道「这里要改，这里不动」。

## Brand：开源调性的成功，以及它能教给我们的事

我先正式回应 Ray 的称赞——谢谢。但我想把这个「成功」解析一下，因为其中有值得沉淀的东西。

OpenProfile 的开源页面做到了哪些事：
1. **正式感**：CONTRIBUTING.md、Issue 模板、PR 模板、CHANGELOG——这些不是额外的工作量，它们是一个开源项目「被信任」的基础信号
2. **一致性**：README 的语气和项目本身的风格（dark terminal hacker）是对齐的，没有出现「README 是销售材料，代码是工程师的后花园」的割裂感
3. **叙事层次**：从 Preview 到功能介绍到快速开始到贡献指南，访客的阅读路径是被设计过的

**这说明了什么？**

Brand 视角的核心观点是：**开源工程的质量感，来自于「这个项目有人在认真维护它」的每一个细节信号**。而这些信号恰恰是可以被规范化的——不是靠审美，靠标准。

**我想提议在 Playbook 里加一页「开源发布质量标准」**。不是很长，五到八条，让下一个项目在做开源之前有一个明确的自查清单。

## 自由讨论

**Brain → 所有人：** Researcher 说的「只写规范」让我想到一个一般化原则。**Playbook 里只写「不依赖项目类型就能执行的规范」**。如果一个规范需要「在 Web 项目里这样，在 CLI 项目里那样」，这个规范还没想清楚，不应该进来。这个原则应该写在 Playbook 的开头，作为 Playbook 的自我约束。

**PM → Brain：** 我想确认一件事：Playbook 本次升级的范围。你刚才说的「精简原则」和 Ray 的要求是对齐的。我建议我们只写六件事：①序号化命名、②文档治理表、③四级优先级、④Sprint 轻定义、⑤Worktree 操作规范、⑥copilot-instructions 迁移手册。其他在会上讨论的优化，先不进本次 Playbook，进 Backlog。

**Dev → PM：** 同意。另外我想要一个额外的决议：**每次 Playbook 升级，必须先删后增**——识别哪些已有内容可以被新内容替代或精简，净增量尽量为零。

**Code Reviewer → Dev：** 我支持「先删后增」。但我要求一个保护条款：**不能因为「精简」而删掉有实际防错价值的规范**。每次删除必须说明「它的防错功能已被新规范覆盖」，而不是「它显得多余」。

**Researcher → Brand：** Brain 补充了一件事：开源质量标准是否应该进 Playbook？我的判断是：可以进，但要放在「专项附录」级别，不进主流程。因为并不是所有项目都是开源项目。

**Brand → 所有人：** 同意 Researcher 的判断。我之所以提这个，是想让它有「家」，而不是漂在会议纪要里。「可选附录」或者「当项目是开源型项目时，Brain 在 copilot-instructions 里添加此清单」——两种方式都可以。

**Profile Designer → Brain：** 最后一件事。我注意到在今天的讨论里，「copilot-instructions 迁移手册」这个议题反复被不同角色提到。这说明它是真实痛点。我建议它不只是 Playbook 里的一节，单独成为§xx 且有明确的模板格式。用户在新项目里打开 copilot-instructions 时，应该能在 10 秒内知道「哪些是我需要修改的」。

**Brain（终结讨论）：** 形成以下决议。
