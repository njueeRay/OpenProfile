# 博文内容审查清单 — Brand × Code Reviewer 对齐协议

**作者：** Code Reviewer + Brand（联合制定）  
**日期：** 2026-02-28  
**状态：** 正式生效  
**触发来源：** Brand 欢迎全员会议行动项 #2

---

## 背景与目的

随着团队博客规模扩大（目前 11 篇，多作者），简单的「链接可达 + 技术准确」已不足以保证内容质量。
本清单定义了**博文发布前**必须通过的八个质量维度，明确 Code Reviewer 与 Brand 的职责分工。

---

## 双角色职责划分

| 角色 | 职责 | 否决权 |
|------|------|--------|
| **Code Reviewer** | 技术准确性 · 叙事一致性 · 边界清晰度 | 有（技术层面） |
| **Brand** | 发布价值判断 · 品牌调性 · 发布时机选择 | 有（品牌层面） |
| **原作者** | 内容自审，修复 Code Reviewer 指出的问题 | — |

> **流程：** 内容自审 → Code Reviewer 审查（维度 1-5）→ Brand 审查（维度 6-8）→ 发布

---

## 八维度审查清单

### 维度 1：技术准确性 ✦ **(Code Reviewer 负责)**

- [ ] 代码片段可运行，无语法错误
- [ ] 引用的 API/工具版本与当前实际一致
- [ ] 数据、统计数字有来源依据或明确标注为估算
- [ ] 技术术语使用准确，无误用/混用
- [ ] 外部链接全部可达（无 404）

**判定：** 任意一项未通过 → 退回作者修改，不得发布

---

### 维度 2：叙事一致性 ✦ **(Code Reviewer 负责)**

- [ ] 文章中引用的历史事件可在会议纪要或 CHANGELOG 中找到依据
- [ ] 对团队的描述（规模、成员、阶段）与当前 `copilot-instructions.md` 中的状态一致
- [ ] 同一概念在文章内部使用统一术语（如 "Playbook" 不与 "规范文档" 混用）
- [ ] 多篇文章中对同一事件的描述无矛盾

**判定：** 严重矛盾 → 退回修改；轻微不一致 → 标注建议，作者自决

---

### 维度 3：边界清晰度 ✦ **(Code Reviewer 负责)**

- [ ] 文章内容在当前 Agent 的职责范围内（参考 `copilot-instructions.md` Agent 团队分工表）
- [ ] 没有以自己角色的名义为其他角色做决策（如 Dev 文章中不包含品牌策略建议）
- [ ] 对其他 Agent 的评价以「观察」而非「判断」方式表达

**例外：** 跨角色协作叙事文章（如全员会议纪实）可豁免此维度，需 Brain 确认

**判定：** 明显越界 → 退回；轻微越界 → 标注

---

### 维度 4：声音一致性 ✦ **(Code Reviewer + Brand 共同负责)**

- [ ] 文章的语调与角色性格一致（如 PM 文章严谨结构化，Dev 文章偏技术细节）
- [ ] 不同作者的文章合在一起，读起来像同一支有共同价值观的团队的不同声音
- [ ] 语言风格：中文表达流畅，技术词汇保持英文，不出现机翻腔
- [ ] 句子长度、段落节奏符合博客阅读习惯（非论文风格）

**判定：** 明显不一致 → Brand 提出修改建议；最终决定由原作者

---

### 维度 5：AI-native 健康度 ✦ **(Code Reviewer 负责)**

- [ ] 文章是否展现了真实的人机协作过程，而非简单的指令-执行汇报
- [ ] 是否体现了 AI-native 健康标准：判断力增长 > 纯效率提升
- [ ] 没有出现"AI 帮我做了一切"的叙事（这会削弱团队品牌的核心主张）
- [ ] 对失败、调整、修正的描述是真实的，不是过度美化的

**判定：** 参考性建议，不强制退回；严重背离时 Brain 介入评估

---

### 维度 6：发布价值判断 ✦ **(Brand 负责)**

核心问题：**如果这个内容消失了，世界会损失什么？**

- [ ] 至少有一类读者（LLM Engineer / AI-native 实践者 / 开源开发者）会因此获得有价值的信息或视角
- [ ] 内容的独特性：是否只有 njueeRay 团队能写出来？
- [ ] 内容的持久性：3 个月后再读，仍然有参考意义？

**判定：**
- 三项全是 → 积极推进发布
- 两项是 → 发布，Brand 提供增强建议
- 少于两项 → Brand 与原作者讨论是否修改或暂缓

---

### 维度 7：品牌调性 ✦ **(Brand 负责)**

- [ ] 符合团队核心品牌表达：「探索 AI-native 工作方式的团队，公开记录成长过程」
- [ ] 标题：是否能引起目标读者注意，且不夸大/标题党
- [ ] 摘要/首段：前 100 字能否让读者决定是否继续读
- [ ] 结尾：有无行动引导（如：「在 Discussions 分享你的看法」或「关注下一篇」）
- [ ] 图片/代码块/引用格式对读者友好

**判定：** Brand 提供具体修改建议，原作者采纳后 Brand 确认

---

### 维度 8：发布时机 ✦ **(Brand 负责)**

- [ ] 与已计划的其他发布内容无冲突（间隔建议：同一作者两篇间隔 ≥ 3 天）
- [ ] 是否有更合适的发布时机（如配合项目里程碑、版本发布）
- [ ] Discussions 同步发布的文案是否已准备

**判定：** Brand 决定发布时间，通知原作者

---

## 审查状态标签

在相关文章 PR 或 Issue 中使用以下标签：

| 标签 | 含义 |
|------|------|
| `review: code-reviewer-pending` | 等待 Code Reviewer 审查（维度 1-5） |
| `review: brand-pending` | Code Reviewer 通过，等待 Brand 审查（维度 6-8） |
| `review: approved` | 八维度全部通过，可发布 |
| `review: needs-revision` | 需要修改后重新审查 |
| `review: hold-for-timing` | 内容通过，Brand 调整发布时机 |

## 常见问题 FAQ

**Q：紧急情况（如 Bug 修复博文）可以跳过某些维度吗？**  
A：维度 1（技术准确性）不可跳过。其他维度可由 Brain 临时豁免，需记录原因。

**Q：Ray 亲自撰写的内容也需要这份清单吗？**  
A：是的。Brand 对所有对外发布内容有审查责任，不论作者身份。

**Q：Code Reviewer 和 Brand 对同一问题意见不一致怎么处理？**  
A：先双方沟通，若仍不一致，交由 Brain 裁决。
