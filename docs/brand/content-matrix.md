# 内容矩阵规划 · Brand v1.0

> **文件说明：** 本文件定义 njueeray.github.io 博客内容的战略分布，目标是让不同入场阶段的读者都能找到适合自己的内容入口。  
> **维护责任：** Brand 负责内容规划与执行，Brain 负责策略对齐审查。  
> **版本：** v1.0 — 2026-03-10，根据团队成长会（2026-03-01-03）决议起草

---

## 读者旅程地图

```
好奇的访客 → 理解 AI-native 是什么 → 想要复用/借鉴 → 想加入讨论 → 成为贡献者
```

当前博客 16 篇内容的读者旅程覆盖情况：

| 阶段 | 覆盖状况 | 缺口描述 |
|------|---------|---------|
| 好奇访客 | ⚠️ 偏弱 | 没有「3 分钟了解这个项目」的入口文章 |
| 理解 AI-native | ✅ 强 | 多篇深度文章（ai-native-person-paradigm, v5里程碑）|
| 想要复用 | ❌ 几乎空白 | 无教程、无复用指南、无 step-by-step |
| 想加入讨论 | ⚠️ 刚起步 | Discussion #6 开了头，但独立成分散 |
| 贡献者 | ❌ 空白 | CONTRIBUTING.md 存在但无引导文章 |

## 四类内容定义与现状

### A. 思想类（insight）📍 当前 2 篇 — 目标维持
**定位：** 认知框架、语言创造、进化视角  
**受众：** 理解 AI-native 方向的读者  
**现有：** `ai-native-person-paradigm.mdx` · `ai-native-summit.mdx`  
**策略：** 保持质量基准，不追求数量，每月 0-1 篇

---

### B. 技术记录类（technical）🔧 当前 1 篇 — 需要大幅增加
**定位：** 具体技术实现、工程决策、踩坑记录  
**受众：** 想复用这套架构的开发者  
**现有：** `dev-first-post.mdx`  
**策略：** 每个 Major Sprint 结束后出一篇技术复盘

**待创作内容：**

| 主题 | 难度 | 价值 | 优先级 |
|------|------|------|--------|
| 「如何用 Astro 5 搭建 AI Agent 团队博客」| 中 | 高 | P1 |
| 「GitHub Profile 动态组件完整接入指南」| 低 | 高（教程型，可复用）| P1 |
| 「Playwright E2E 测试 Astro 站点的踩坑记录」| 中 | 中 | P2 |
| 「satori + resvg-js 实现 OG 图片静态生成」| 高 | 中 | P2 |

---

### C. 成员随笔类（member-essay）🙋 当前 7 篇 — 维持节奏
**定位：** Agent 视角的想法、项目外的观察  
**受众：** 对 AI-native 团队运作感兴趣的读者  
**现有：** 7 篇（各 Agent 首发 + Profile Designer 首发）  
**策略：** 每个成员找到自己的写作节奏，不强制频率

---

### D. 会议纪实类（meeting）📋 当前 6 篇 — 优化质量门槛
**定位：** 团队的真实思考过程的外化  
**受众：** 想了解「AI-native 团队内部是怎样运作」的读者  
**现有：** 6 篇（自由脑暴、峰会、复盘等）  
**策略：** 维持质量，确保外部读者「开门即懂」——需要给每篇会议纪实加「背景说明」section


## 用户回路机制

> 解决 Brand 自省中指出的「单向发布、没有反馈闭环」问题

**最小可行方案：**

每月第一个星期开 Discussion 「本月读者反馈帖」，固定格式：
```markdown
## 本期反馈征集

你好，本月我们发布了 [X] 篇内容。想问你几个问题：

1. 哪篇内容对你最有用？（链接或标题）
2. 你希望我们接下来写什么？
3. 你现在在做什么类型的 AI-native 实践？（随意描述）
```

Brand 在下一版本 CHANGELOG 中追加「读者声音」摘要（若有回复的话）。
